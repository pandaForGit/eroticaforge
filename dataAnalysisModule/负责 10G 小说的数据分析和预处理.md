根据我们之前的计划（尤其是向量数据库设计和 10G 小说合集的利用），**后端确实需要一个专门的模块来负责 10G 小说的数据分析和预处理**。

这个模块不是可有可无，而是**非常必要**，它能把 10G 原始数据转化为项目真正可用的高质量资源（人物卡、专题参考库、元数据标签等），避免直接把海量噪声塞进向量数据库。

### 为什么必须有这个专用模块？

1. **10G 数据量太大，不能直接扔进 RAG**  
   全量索引会产生海量向量，内存/磁盘占用爆炸，检索时噪声极大，导致生成质量下降。

2. **核心目标是“精准参考”而不是“全部塞进去”**  
   我们需要从 10G 中**提取**有价值的部分：人物卡、关键情节总结、题材标签、风格片段等。

3. **后续使用需要结构化数据**  
   只有经过分析、分类、提取后，才能实现“新建 NTR 故事时只加载 NTR 参考库”这样的精准控制。

### 推荐模块设计：**DataAnalysisModule**（或叫 **CorpusProcessorModule**）

#### 模块职责（清晰划分）

- **小说分类**：按题材（NTR、SM、二次元、现代、脑洗等）分类
- **人物卡提取**：从每本小说中自动提取主要人物，并生成高质量人物卡
- **元数据打标**：为每本小说/章节/人物添加标签（theme、intensity、style 等）
- **总结生成**：为长篇小说生成章节摘要或整书摘要
- **参考库构建**：按题材生成可按需加载的专题向量子库
- **数据清洗**：去除广告、水印、无用章节等噪声

#### 建议模块结构（包路径）

```
application/
  └── corpus/
      ├── service/
      │   ├── NovelClassifierService.java          # 分类主服务
      │   ├── CharacterCardExtractorService.java   # 人物卡提取
      │   ├── CorpusSummarizerService.java         # 总结生成
      │   └── ReferenceLibraryBuilderService.java  # 构建专题参考库
      ├── task/
      │   └── BatchCorpusProcessor.java            # 批量处理任务（支持断点续跑）
      ├── model/
      │   └── ClassificationResult.java
      └── config/
          └── CorpusProcessingConfig.java
```

**当前代码位置（与主后端解耦）**：独立 Maven 工程 `dataAnalysisModule/pom.xml`（artifact：`eroticaforge-data-analysis`），实现包为 `com.eroticaforge.dataanalysis`，已含清洗与 `NovelClassifierService`（输出 JSONL）。主工程 `eroticaforge` 不再包含上述语料批处理代码。

#### 技术实现建议

- **批量处理**：使用 Spring Batch 或简单多线程 + 任务队列（推荐 Spring Batch，支持断点、重试、监控）
- **LLM 调用**：使用 4B 或 7B 模型（速度快、显存友好），避免用 9B
- **存储**：分类结果和提取的人物卡先存到本地 JSON / 数据库，再决定是否导入向量库
- **进度控制**：支持暂停、继续、进度监控（很重要，10G 处理时间会很长）

### 开发优先级建议

1. **先实现 NovelClassifierService**（你已经有基础代码，可以继续完善）
2. **然后做 CharacterCardExtractorService**（从小说中批量提取人物卡，这是高价值输出）
3. **再做 ReferenceLibraryBuilderService**（按题材构建可加载的参考库）

这样一步步来，不会一下把项目搞得太重。
