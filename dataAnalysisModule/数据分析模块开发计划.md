**文档版本**：1.0  
**更新日期**：2026-03-27

## 2. 目标与非目标

### 2.1 目标

- **分类**：按主类型、标签、风格、强度等为每本（或每文件）产出结构化元数据，供后续过滤与专题库构建。  
- **提取**：人物卡、章节/全书摘要等**高价值片段**，而非全文向量化。  
- **清洗**：剔除明显噪声（过短文件、广告段落等），规则可配置。  
- **批处理**：支持长时间运行、**断点续跑、进度可观测、失败可重试**（与 `负责 10G 小说的数据分析和预处理.md` 中 Spring Batch 或队列方案一致，具体选型在阶段 1 敲定）。  
- **落库路径清晰**：中间结果（JSON/表）→ 审核或抽样校验 → 再导入 `type=reference` 向量层（见向量库设计文档）。

### 2.2 非目标（首版可不实现）

- 替代主应用内的在线生成、StoryState、用户故事 RAG。  
- 多用户鉴权与分布式集群调度（单机批处理优先）。  
- 对 10G 全文无差别嵌入索引。

## 3. 与主工程的职责划分

| 层级 | 位置 | 说明 |
|------|------|------|
| 语料清洗 / 分类（当前已实现） | `dataAnalysisModule/src/main/java/com/eroticaforge/dataanalysis/corpus/` | `CorpusSanitizer`、`NovelClassifierService`、可选 `CorpusBatchApplicationRunner`；配置见同模块 `application.yml` 中 `erotica.corpus` |
| LLM（仅 Chat） | `dataAnalysisModule/.../support/OpenAiChatCompletionClient.java` | 直连 `POST /v1/chat/completions`，避免 LangChain4j 与 llama-server 返回格式不一致导致空文本 |
| 主后端 | `ai-writer/src/main/java/com/eroticaforge/` | 故事、RAG、持久化等；**不包含**语料批处理代码 |
| 后续落库 / 参考库构建 | 主工程 `CorpusJsonlReferenceImporter` + `RagIngestionService#ingestCorpusReference` | 见下文「接入主工程 RAG」 |
| 文档 | `dataAnalysisModule/*.md` | 计划、数据契约、变更记录 |

**构建与运行（独立模块）**：在仓库根目录执行  
`mvn -f dataAnalysisModule/pom.xml test`  
批量分类（需 llama-server，默认读 `./src/main/resources/小说`，写出 `./out/*.jsonl`）：  
`mvn -f dataAnalysisModule/pom.xml spring-boot:run -Dspring-boot.run.profiles=corpus-run`  
环境变量：`CORPUS_INPUT`、`CORPUS_OUTPUT_JSONL` 可覆盖输入/主输出路径；主工程端口 `8090` 与本模块 `8092`（`DATA_ANALYSIS_SERVER_PORT`）错开。

### 接入主工程 RAG（`src`）

1. 得到 `corpus_classification.jsonl` 后，在主应用（JDK + DB + Ollama 嵌入）中注入 `CorpusJsonlReferenceImporter`，调用  
   `importFromJsonl(Paths.get(".../corpus_classification.jsonl"), Paths.get("语料根目录"), ragProperties.getReferenceCorpusStoryId())`。  
   `语料根目录` 须与数据分析批处理时的 **input 根** 一致，以便 `source_relative_path` 能拼出真实 txt。  
2. `application.yml`：`erotica.rag.include-reference-corpus: true` 后，`RagRetrievalService` 会把参考库切块与**当前故事**的切块一起向量检索并按分数取 top-k（段落标签为「参考」）。  
3. 参考库使用的 `story_id` 默认为 `system-corpus-reference`，勿与普通故事 ID 冲突。

## 4. 建议服务拆分（与现有构思一致）

以下名称可与 `负责 10G 小说的数据分析和预处理.md` 对齐，实现时按依赖顺序增量交付。

1. **NovelClassifierService**（P0）  
   - 输入：文件路径或文本前缀（如前 3000～5000 字）。  
   - 输出：结构化 JSON（字段与 `docs/product/小说进行分类.md` 对齐，可扩展 `source_file`、哈希）。  
   - 验收：小样本集上 JSON 可解析率、人工抽检标签合理性。

2. **CorpusSanitizer / 清洗步骤**（P0，可并入分类流水线）  
   - 过短跳过、编码检测、简单广告规则。  
   - 验收：日志中可统计跳过原因。

3. **CharacterCardExtractorService**（P1）  
   - 从选定章节或抽样片段提取人物卡，输出与 `高质量人物卡` 指南一致或可映射的 schema。  
   - 验收：抽样对比人工整理卡片的完整度。

4. **CorpusSummarizerService**（P1）  
   - 章节级或书本级摘要，供后续 chunk 与 metadata。  
   - 验收：摘要长度与主题一致性（规则 + 抽检）。

5. **ReferenceLibraryBuilderService**（P2）  
   - 按 `theme` / `tags` 将已通过校验的文本片段送入嵌入与 `erotica_rag_chunks`（`type=reference`），metadata 与向量库设计文档一致。  
   - 验收：给定标签过滤检索命中预期专题片段。

6. **BatchCorpusProcessor**（横切）  
   - 编排上述步骤；持久化进度（文件或表）；支持 resume。  
   - 验收：中断后继续无重复处理或具备幂等键。

## 5. 分阶段实施计划

### 阶段 A：契约与骨架（约 1～2 个迭代）

- [ ] 定义分类 JSON schema 与版本号（文档 + Java record/DTO）。  
- [x] 在 `dataAnalysisModule` 独立工程中建立 `com.eroticaforge.dataanalysis.corpus` 与 `OpenAiChatCompletionClient`（与主工程共用 `langchain4j.open-ai` 配置键指向 llama-server）。  
- [ ] 实现 **单文件** 端到端：读盘 → 调用 ChatModel → 写 JSON 一行/一条。  
- **验收**：对 10～20 个 txt 跑通，无 Spring 循环依赖。

### 阶段 B：批量与可恢复（P0 完成度）

- [ ] 接入目录遍历 + 进度文件或表（`processed_file_hash` / `offset`）。  
- [ ] 限流与并发（避免拖垮本机 llama/Ollama）。  
- [ ] 结构化日志：`storyId` 不适用处用 `sourcePath` + `correlationId`。  
- **验收**：故意中断后可续跑；错误样本单独落盘便于重放。

### 阶段 C：人物卡与摘要（P1）

- [ ] Prompt 与输出 schema 落地（参考 `完整 Prompt 模板.md` 若已有分类/提取模板则复用）。  
- [ ] 输出写入中间存储（JSONL 或表），**不默认**全开向量索引。  
- **验收**：与产品文档中的「参考库」场景可手动选用提取结果。

### 阶段 D：专题参考库入库（P2）

- [ ] 与 `RagIngestionService` 或专用 ingester 对接，写入 Layer 3 约定 metadata。  
- [ ] 维度与 `pgvector` 列一致（见 `后端代码计划文档` 阶段 3.1）。  
- **验收**：MetadataFilter 按 `theme`/`tags` 过滤检索有效。

### 阶段 E：质量与运维

- [ ] 单元测试：JSON 解析、路径边界、断点逻辑。  
- [ ] 可选集成测试：Testcontainers PostgreSQL（团队允许时）。  
- [ ] 回写：`docs/api/API 接口定义.md` 若增加管理端接口则标注实现状态。

## 6. 技术约束（摘录）

- **LLM**：优先使用项目已配置的**小参数量**总结/分类模型（如 Qwen 4B 量级），与 `后端代码计划文档` 一致；避免默认使用过大模型导致批处理过慢。  
- **嵌入**：仅在建参考库步骤使用 **Ollama bge-m3**，维度与库表锁定一致。  
- **存储**：中间产物允许本地 JSON/JSONL；正式 metadata 建议进 PostgreSQL，与向量设计方案一致。

## 7. 风险与缓解

| 风险 | 缓解 |
|------|------|
| 10G 处理时间过长 | 断点续跑、可配置并发、先分类后精选入库 |
| 模型输出非合法 JSON | 重试、降级模板、失败样本隔离 |
| 向量库膨胀 | 只索引摘要与精选片段；严格执行 `type=reference` 与清理策略 |
| 隐私与内容安全 | 本地部署、路径不提交版本库；见 `docs/security/安全与隐私说明.md` |
