## 1. 目标与约束

### 1.1 本计划要解决什么

在现有文档已定义**产品能力、接口、分层架构、PostgreSQL + pgvector 数据模型、Prompt 体系**的前提下，给出**可执行的后端实现顺序**：每一步做什么、产出什么、如何验收，便于单人或 Agent 按阶段推进。

### 1.2 技术选型（与当前文档对齐）

| 层级 | 选型 | 说明 |
|------|------|------|
| 运行时 | Java 21 + Spring Boot 3 | 与 PRD / 架构文档一致 |
| LLM 调用 | **llama.cpp**（`llama-server` OpenAI 兼容 API） | 见 `本地 NSFW 小说制作搭建计划--java.md`；LangChain4j 使用 `langchain4j-open-ai` 指向 `base-url: http://localhost:8081/v1`（端口可配置） |
| 嵌入模型 | **Ollama + bge-m3** | 见 `模型管理与配置.md`；`langchain4j` 中 `ollama.base-url` + `embedding.options.model: bge-m3` |
| 主生成 / 总结 | **Qwen 4B** 量级 GGUF（多步链时可顺序调用或分实例） | 硬件以 GTX 1660 6GB 为基准 |
| 向量与业务数据 | **PostgreSQL + pgvector** | 见 `完整 PostgreSQL 与 pgvector 数据模型.md`；**不再使用 Redis** 作为向量库或主存储 |
| RAG / 链 | LangChain4j（Easy RAG、AiServices 可选） | 向量存储使用 pgvector 适配实现 |

### 1.3 不在本计划内首次完成的内容（可标为后续迭代）

- 多用户鉴权（JWT / API Key）— API 文档已预留  
- 图生文、JoyCaption 等 v2 能力 — 见 `未来路线图与待办事项.md`  
- 完整前端 UI — 后端以 REST + SSE 为先

## 2. 总体阶段划分

```text
阶段 0 环境与仓库
    ↓
阶段 1 工程骨架与配置（Spring + LangChain4j + llama.cpp + Ollama embedding + PostgreSQL）
    ↓
阶段 2 领域模型与持久化层（JPA/JDBC、pgvector 表）
    ↓
阶段 3 RAG：摄入与检索（无生成或 Mock 生成）
    ↓
阶段 4 生成核心：单模型流式 + PostGeneration + StoryState 更新
    ↓
阶段 5 REST API 对齐《API 接口定义》（故事、文档、生成、章节、state）
    ↓
阶段 6 多模型链（可选总结步）+ Lorebook + 健康检查
    ↓
阶段 7 测试、日志、错误处理与文档回写
```

以下各节按**步骤**展开，每步包含：任务、涉及文档、建议产出、验收标准。

---

## 3. 阶段 0：环境与仓库

### 步骤 0.1 本地依赖就绪

- **任务**：安装 JDK 21、Maven、NVIDIA 驱动/CUDA（若用 GPU）；安装并启动 **PostgreSQL（含 pgvector 扩展）**；安装 **Ollama** 并 `ollama pull bge-m3`；编译运行 **llama.cpp** 的 `llama-server`，确认 OpenAI 兼容端可访问。
- **参考**：`docs/deployment/安装部署指南.md`、`docs/development/模型管理与配置.md`。
- **产出**：本机可手动调用 chat completions（llama）与 embedding（Ollama bge-m3）。
- **验收**：`psql` 中 `CREATE EXTENSION vector` 成功；嵌入请求返回固定维度向量。

### 步骤 0.2 代码仓库初始化

- **任务**：建立 Maven 工程，`groupId`/`artifactId` 与 `项目目录结构说明.md` 中建议包名一致（如 `com.eroticaforge`）；`README.md` 说明如何启动 Postgres、llama-server、Ollama。
- **产出**：空 Spring Boot 可 `mvn spring-boot:run`。
- **验收**：应用启动无报错，健康占位接口（如 `/actuator/health` 或自定义 `/api/health` 骨架）可访问。

## 4. 阶段 1：工程骨架与配置

### 步骤 1.1 依赖与分层目录

- **任务**：按 `项目目录结构说明.md` 创建包：`config`、`domain`、`application/service`、`application/dto`、`infrastructure`、`presentation/controller`、`utils`。
- **任务**：在 `pom.xml` 中加入 LangChain4j 核心、`langchain4j-open-ai`、`langchain4j-ollama`（用于 **EmbeddingModel**）、PostgreSQL 驱动、Spring Data JDBC 或 JPA、pgvector 相关适配依赖（按所选集成方式）。
- **产出**：目录与空类/接口占位（避免循环依赖）。
- **验收**：编译通过；包依赖关系符合架构文档（domain 无 Spring 注解）。

### 步骤 1.2 LLM Bean：对接 llama.cpp

- **任务**：在 `config` 中配置 `OpenAiChatModel` / `OpenAiStreamingChatModel`，`base-url` 指向 llama-server。
- **参考**：`本地 NSFW 小说制作搭建计划--java.md`。
- **产出**：`LlmConfig.java`（名称可自定）+ `application.yml` 外部化配置。
- **验收**：单元测试或 `@SpringBootTest` 中调用一次 `generate("ping")` 有返回。

### 步骤 1.3 嵌入模型：Ollama bge-m3

- **任务**：按下列结构配置 **EmbeddingModel**（具体属性名以 LangChain4j 版本为准）：

```yaml
langchain4j:
  ollama:
    base-url: http://localhost:11434
    embedding:
      options:
        model: bge-m3
```

- **产出**：可注入的 `EmbeddingModel` Bean；维度与 `pgvector` 列一致（常见 **1024**，以实测为准）。
- **验收**：对短文本 `embed` 得到固定维度向量。

### 步骤 1.4 PostgreSQL 与 pgvector

- **任务**：配置数据源；创建/迁移表结构（见 `完整 PostgreSQL 与 pgvector 数据模型.md`）；实现或接入 LangChain4j **pgvector** `EmbeddingStore`。
- **产出**：`PostgresConfig` / Flyway 脚本 + 测试连接。
- **验收**：应用启动时可写入并查询 `erotica_rag_chunks`（或等价表）。

## 5. 阶段 2：领域模型与持久化基础

### 步骤 2.1 领域 POJO

- **任务**：实现 `StoryState`、`Story` 元信息、`Chapter` 等与 `完整 PostgreSQL 与 pgvector 数据模型.md` 字段对齐的模型；版本号 `version` 用于乐观合并。
- **参考**：`核心 Service 实现指南.md` 中 StoryState 推荐字段。
- **产出**：`domain/*.java`。
- **验收**：与表结构/JSONB 序列化一致。

### 步骤 2.2 仓储门面

- **任务**：`infrastructure/persistence` 中封装故事元数据、章节列表、`story_states`、文档元数据的读写；向量表由 `EmbeddingStore` + 元数据表协同管理。
- **产出**：`NovelRepository` / `StoryStateRepository` 等（命名自定）。
- **验收**：写入再读出 `story_states` 无字段丢失。

## 6. 阶段 3：RAG 摄入与检索

### 步骤 3.1 嵌入与维度

- **任务**：**bge-m3** 维度与 `vector(N)`、`langchain4j` store 配置一致。
- **参考**：`核心 Service 实现指南.md` 3.1–3.2。
- **验收**：对短文本 `embed` 得到固定维度向量。

### 步骤 3.2 `RagIngestionService`

- **任务**：实现 `ingestDocument` / `ingestText`：切分、嵌入、`EmbeddingStore.addAll`（pgvector）；文档元数据写入 `erotica_documents`（或等价表）。
- **参考**：`核心 Service 实现指南.md` 3.1（pgvector 存储）。
- **验收**：上传测试 txt 后，Postgres 中可检索到向量命中。

### 步骤 3.3 `RagRetrievalService`

- **任务**：实现 `retrieveRelevantContext(query, state)`：top-k、相似度阈值、格式化为带标签上下文；多模型模式下保留原始 chunks 供总结步使用。
- **参考**：`核心 Service 实现指南.md` 3.2。
- **验收**：给定固定索引数据，检索结果稳定可复现。

## 7. 阶段 4：生成核心与后处理

### 步骤 4.1 `PromptTemplates` 与构建管线

- **任务**：将 `docs/prompts/完整 Prompt 模板.md` 落地为 `utils/PromptTemplates.java`（或资源文件）。
- **产出**：纯字符串模板 + 统一 `buildFullPrompt(...)` 入口。
- **验收**：单元测试对固定输入快照比对 Prompt 片段（可选）。

### 步骤 4.2 `NovelGenerationService`（单模型优先）

- **任务**：先实现**单模型**：`RAG + StoryState + Lorebook（可先空实现）+ userPrompt → StreamingChatModel`。
- **参考**：`核心 Service 实现指南.md` 3.4；流式 Bean 对接 llama.cpp。
- **产出**：`NovelGenerationService` + 流式 API 封装。
- **验收**：SSE 或阻塞式能输出一段连贯文本。

### 步骤 4.3 `StoryStateService`

- **任务**：`getCurrentState`、`updateState` / `mergeDelta`；生成后用 4B 模型或规则生成结构化摘要。
- **参考**：`核心 Service 实现指南.md` 3.3。
- **验收**：每轮生成后 `version` 递增，数据库中 `state` 更新。

### 步骤 4.4 `PostGenerationService`

- **任务**：顺序：分配 `chapterId`、写入章节、更新故事元信息、更新 `StoryState`、可选将新章节摘要再索引入 RAG。
- **参考**：`核心 Service 实现指南.md` 3.5；PostgreSQL 数据模型文档。
- **产出**：单次生成闭环可持久化。
- **验收**：重启应用后章节与 state 仍在。

## 8. 阶段 5：REST API 对齐

按 `docs/api/API 接口定义.md` 逐项实现（建议顺序）：

| 顺序 | 方法 | 路径 | 说明 |
|------|------|------|------|
| 1 | GET | `/api/health` | PostgreSQL、llama-server、Ollama（embedding）可达性 |
| 2 | POST/GET/DELETE | `/api/stories` | 故事 CRUD |
| 3 | POST | `/api/stories/{storyId}/documents` | multipart 上传 → Ingestion |
| 4 | POST | `/api/stories/{storyId}/generate/stream` | SSE，`useMultiModel` 先透传或固定 false |
| 5 | POST | `/api/stories/{storyId}/generate` | 非流式调试 |
| 6 | GET | `/api/stories/{storyId}/chapters`、`.../chapters/{chapterId}` | |
| 7 | GET/PUT | `/api/stories/{storyId}/state` | |
| 8 | GET/POST | `/api/lorebook` | 可与 `LorebookService` 同步 |

- **任务**：统一错误响应格式（见 API 文档第 4 节）；限制上传大小（如 10MB）。
- **验收**：Postman/curl 可跑通主流程：建故事 → 上传文档 → 流式生成 → 拉章节与 state。

## 9. 阶段 6：多模型链与增强

### 步骤 6.1 多模型链（`useMultiModel=true`）

- **任务**：`RagRetrieval → Summarizer（4B，低温度）→ 主生成（4B，高温度）`；显存紧张时顺序调用。
- **参考**：`核心 Service 实现指南.md` 第 4 节；`多模型协作.md`。
- **验收**：开启多步后 StoryState 与正文质量优于单步的基准用例（可人工评估）。

### 步骤 6.2 `LorebookService`

- **任务**：关键词匹配 `用户输入 + state 摘要`，拼接触发描写进 Prompt。
- **参考**：`完整 PostgreSQL 与 pgvector 数据模型.md` 中 Lorebook 表。

### 步骤 6.3 防死循环

- **任务**：采样参数 + Prompt 规则；可选简单后处理检测重复 n-gram。
- **参考**：PRD、`核心 Service 实现指南.md` 6。

## 11. 风险与依赖

| 风险 | 缓解 |
|------|------|
| 嵌入维度与 pgvector 列不一致 | 阶段 1.3 与 3.1 用一次 `embed` 固定 `N` |
| 多模型并发 OOM | 默认顺序调用；文档中写清 `llama-server` 实例数与端口 |
| LangChain4j 与 pgvector 集成差异 | 锁定版本并参考官方示例 |
