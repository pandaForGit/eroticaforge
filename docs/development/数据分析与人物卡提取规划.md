# 数据分析与人物卡提取规划

**文档版本**：1.0  
**更新日期**：2026-03-27  
**关联**：`dataAnalysisModule/`（独立批处理工程）、`docs/guides/高质量人物卡.md`、`docs/deployment/EroticaForge 项目量身定制的向量数据库设计方案.md`

---

## 1. 你的理解是对的

**语料数据分析**不止「分类 + 参考库」，还应包含 **人物卡分析（自动提取 / 结构化）**：从 10G 原文中捞出**可复用的角色设定**，与 `高质量人物卡.md` 中的标准对齐或映射，供：

- 用户一键导入 / 二次编辑后写入正式人物卡库；  
- 后续 RAG 按 `character_names`、题材标签做更细召回（与向量库设计中的 metadata 方向一致）。

当前 **已实现**：清洗、小说级分类 JSONL、主工程侧参考 txt 入库 RAG（见 `CorpusJsonlReferenceImporter`）。  
**规划中**：人物卡批量提取服务与输出契约、与主工程持久化的衔接。

---

## 2. 流水线在整体中的位置

```text
原始 txt（GBK/UTF-8）
    → 清洗（CorpusSanitizer，已实现）
    → 小说级分类 JSONL（NovelClassifierService，已实现）
    → 【P1】人物卡提取 JSONL/表（CharacterCardExtractor，待实现）
    → 【P1】可选：章节/全书摘要（CorpusSummarizer，待实现）
    → 人工或规则抽检
    → 主工程：人物卡导入 API / 表结构 + 可选向量片段（reference / character_card）
```

人物卡提取 **依赖** 分类结果时可先做「按标签筛批次」（例如只处理 `tags` 含目标题材的文件），降低无效 LLM 调用；也可与分类解耦、仅按目录分批。

---

## 3. 人物卡提取（规划）— 需求要点

| 维度 | 说明 |
|------|------|
| **输入** | 单本或抽样片段（章节头尾、对话密集段、全文分段摘要后再提取等，具体策略在实现时定）；须复用与分类一致的 **编码检测（UTF-8 / GBK）**。 |
| **输出** | 结构化 JSON（每角色一条或每文件多条），字段与 `docs/guides/高质量人物卡.md` **可映射**：基础信息、性格、说话风格、性癖与偏好、心理与背景、触发反应等；额外附 `source_relative_path`、`novel_title`、`classification_tags`（可选）便于追溯。 |
| **模型** | 与分类相同链路：本机 **llama-server**，小参数量、**控制单次 Prompt 长度**（与分类模块已踩坑的上下文策略一致）。 |
| **批处理** | 与分类一致：**串行或弱并发**、**断点续跑**、失败行单独记录；输出 **JSONL** 优先，确认后再导入 DB。 |
| **实现位置** | 建议仍在 **`dataAnalysisModule`**（`com.eroticaforge.dataanalysis`）新增 `CharacterCardExtractorService`（及 Prompt、DTO），避免主工程与 10G 批处理强耦合。 |

---

## 4. 与主工程（`src`）的衔接（规划）

| 步骤 | 说明 |
|------|------|
| 导入 | 主工程增加「从 JSONL / 批量 JSON 导入人物卡」的 **Application Service + API**（或管理端），写入未来人物卡表或与现有 `CharacterCard` 领域模型对齐。 |
| RAG | 可选：将「单卡短描述 + 关键词」嵌入 pgvector，`metadata.type = character_card`（与向量库设计文档 Layer 3 思路一致）；检索时与 `story_id`、标签联合过滤（需迭代 `RagRetrievalService`）。 |
| 与参考正文库关系 | **参考 txt 全文切块**与 **人物卡向量**分工：前者供文风/情节参考，后者供角色一致性；两者可共用 `reference` 类 story_id 或分库存储，按产品选择。 |

具体表结构以 `docs/deployment/完整 PostgreSQL 与 pgvector 数据模型.md` 及后续人物卡表设计为准。

---

## 5. 推荐实现顺序（与 `数据分析模块开发计划.md` 一致）

1. **P0（已有）**：分类 JSONL + 主工程参考库导入。  
2. **P1**：`CharacterCardExtractorService` + 输出 schema 定稿（对照 `高质量人物卡.md`）。  
3. **P1**：`CorpusSummarizerService`（辅助长文分段提取质量）。  
4. **P2**：主工程导入 + 可选人物卡向量与检索过滤。

---

## 6. 验收建议（人物卡阶段）

- 小样本（10～20 本）抽取后，与人工整理卡对比 **字段完整度、OOC 风险、重复角色合并**。  
- JSON Schema 版本号固定，便于后续迁移与前端表单绑定。
