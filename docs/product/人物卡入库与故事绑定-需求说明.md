# 人物卡入库与故事绑定 — 需求说明

**文档版本**：1.2  
**日期**：2026-04-03  
**状态**：业务已确认（含：创建故事后可改绑定；性格 = DB + 模板分节）  
**关联文档**：`docs/planning/人物卡与快照-实现计划.md`、`docs/architecture/人物卡与RAG编排业务流程.md`、`docs/product/人物卡导入与故事绑定需求说明.md`、`docs/development/数据分析与人物卡提取规划.md`、`dataAnalysisModule/out/character_cards.jsonl`

## 1. 背景与目标

数据分析模块已能产出 **人物卡 JSONL**（每行对应一篇源文本的抽取结果，内含多名角色结构化字段）。主应用（EroticaForge）当前：

- 故事创建仅包含 **标题、标签**（`CreateStoryRequest`），**未绑定**独立人物卡实体；
- 生成链路为 **StoryState + RAG 召回 + Lorebook + 用户输入**（`PromptComposer` / `NovelGenerationService`），**没有**「从人物卡库注入固定人设」的独立区块。

**本需求目标**：

1. 将 `character_cards.jsonl`（及后续同 schema 文件）**导入数据库**，形成可查询、可复用的人物卡库。  
2. **新建故事**时从人物卡库 **多选**；服务端为每个选中库卡在故事下写入 **快照行**（深拷贝）。用户在故事内 **可改** 名称、性格等，**不影响**人物卡库母卡。  
3. **创作过程中**可在故事内 **新增** 人物快照（非自库克隆），与克隆条目一并参与生成。  
4. **每次调用模型生成**时，按故事下快照 **当前内容** 写入提示词，与 **RAG 召回正文** 一起供模型参考。  

> 说明：「RAG 时把人物性格给到提示词」在业务上指 **同一次生成请求内**，模型同时看到 **向量召回片段** 与 **人物卡人设段落**；人设段落优先以 **确定性注入**（见业务设计文档）实现，是否与人物卡向量检索合并可作为二期能力。

## 2. 数据源约定（JSONL）

输入文件与 `dataAnalysisModule` 产出对齐，**单行** 为合法 JSON，核心结构示例（字段以实际文件为准）：

| 概念 | 说明 |
|------|------|
| `schema_version` | 人物卡 schema 版本，导入需校验或迁移 |
| `characters[]` | 该源文件抽取出的多名角色 |
| 角色字段 | 如 `name`、`age`、`identity`、`appearance`、`personality`、`background`、`triggers[]`、`nsfw_profile`、`psychology_and_relations`、`sample_dialogues[]` 等 |
| `source_relative_path` | 源语料相对路径，追溯用 |
| `content_sha256` | 源内容哈希，可用于 **幂等导入 / 变更检测** |

**约束（已确认）**：不同源文件可出现 **同名不同角色**；必须以 **稳定主键** 区分（不能只按 `name` 唯一）。**故事绑定采用快照**：编辑故事内人设不反向更新库卡。

## 3. 用户场景（User Stories）

1. **导入**：作为运营/作者，我能从指定 JSONL 路径执行导入（或管理端上传），导入后能在列表中浏览、搜索人物卡。  
2. **幂等与更新**：重复导入同一 `content_sha256` 时行为可预期（跳过 / 覆盖 / 版本递增），不产生无限重复行。  
3. **建故事选人**：创建故事时从库 **多选**，系统生成 **快照**；我能在故事详情里 **改快照** 或 **删快照**。  
4. **中途加人**：我能 **新增** 仅属于本故事的人物快照。  
5. **生成**：续写时 Prompt 含 **当前快照列表** 的格式化人设，并与 RAG、StoryState、Lorebook 共存。  

---

## 4. 功能范围

### 4.1 本期建议范围（MVP）

- DB：**人物卡库表**（导入母卡）+ **故事人物快照表**（`story_id` + 可编辑 `payload` + 可选 `cloned_from_library_id`）。  
- 导入：JSONL → 人物卡库（幂等策略与 skipped 报告）。  
- API：库列表/搜索；`POST /api/stories` 支持传 **库卡 ID 列表** 以创建快照；`GET/PATCH` 故事下快照（增删改排序）。  
- 前端：新建故事 **多选库卡**；故事详情 **快照列表与编辑**（可新增空快照）。  
- 生成：`PromptComposer` 增加占位符（如 `{{characterProfiles}}`），内容仅来自 **快照表**（不依赖向量检索）。  

### 4.2 二期（可选，文档预留）

- 将快照摘要 **写入向量库**参与检索，与正文 RAG 混合排序。  
- 库卡侧：合并重复角色、批量导出、与 `docs/guides/高质量人物卡.md` 完整字段校验。

## 5. 非功能需求

| 项 | 说明 |
|----|------|
| 性能 | 单故事绑定人物数建议上限（如 8～16），提示词长度需可配置截断 |
| 隐私与安全 | JSONL 可能含敏感内容，导入与 API 访问范围与现有故事数据一致 |
| 可观测性 | 导入批次数、跳过/失败行数、绑定故事数可日志或简单指标 |

---

## 6. 验收标准（建议）

1. 给定样例 `character_cards.jsonl`，执行导入后 DB 中记录数与策略一致（按角色条数或按源文件条数，以业务设计为准）。  
2. 创建故事时提交 N 个有效 **库卡** ID，详情返回 N 条 **快照**；改快照后库卡不变。  
3. 新增一条故事内快照后，生成 Prompt 中含该人设。  
4. 生成 Prompt 中人物块与 **快照表当前数据**一致。  
5. 删除故事时快照级联删除；删除库卡**不**自动删除已存在故事中的快照（或产品另定策略，需显式文档）。  

---

## 7. 待实现前技术拍板（非业务待定）

1. **入库粒度**：推荐 **每个库角色一行**（附 `source_relative_path`、`content_sha256`、行内序号），便于搜索与克隆。  
2. **删除库卡时**：已克隆到故事的快照是否保留 —— 默认 **保留**（快照已脱钩）；若需「库删则快照失效」另加开关。  
3. **提示词长度**：多快照时压缩/截断策略与配置项。  
4. **UI**：库列表展示 `source_relative_path` 片段以区分同名。  
5. **导入入口**：启动导入 / 管理 API / CLI，以实现阶段定。
