# 人物卡导入与故事绑定 — 需求说明

**文档版本**：0.3  
**更新日期**：2026-04-03  
**产品已确认**：① 同名角色、不同源文件 = **不同库卡**；② **创建故事后**仍可增删改绑定（快照 CRUD）；③ 「RAG 时给性格」= **快照从 DB 读出 + Prompt 模板独立分节**（`{{characterProfiles}}`），不经向量；故事侧为**可编辑快照**，支持中途新增。  
**实现排期**：见 `docs/planning/人物卡与快照-实现计划.md`。  
**关联代码与文档**：

- 人物卡产出：`dataAnalysisModule/out/character_cards.jsonl`（每行一条 JSON，内含 `characters[]`、`schema_version`、`source_relative_path`、`content_sha256`）
- 现有生成链路：`PromptComposer`、`NovelGenerationService`、`RagRetrievalService`
- 已有规划：`docs/development/数据分析与人物卡提取规划.md`、`docs/guides/高质量人物卡.md`

## 1. 背景与目标

数据分析模块已能批量产出结构化人物卡 JSONL。主应用侧需要：

1. **将人物卡持久化到数据库（人物卡库）**，便于列表、搜索与幂等导入；**同一 `name` 若 `source_relative_path`（及稳定主键）不同，视为不同卡**，永不自动合并。  
2. **新建故事时从人物卡库多选**：服务端为每个选中项在故事下插入一条**快照行**（深拷贝当时字段）；之后作者在故事内改名称、性格等，**只改快照，不改库中母卡**。  
3. **创作过程中**可在故事内**新增**人物快照（无母卡，`source_library_id` 为空），与从库克隆的条目一视同仁参与生成。  
4. **每次调用大模型生成时**，按故事下快照的**当前内容**拼装「人物设定」段落（与 RAG 召回并列），减少 OOC。

> 说明：「RAG 时把人物性格给到提示词」指**同一次生成请求内**，模型同时看到 **向量召回的语料片段** 与 **本故事人物快照**；快照内容**确定性注入**，不把「性格」本身再依赖向量检索（除非后续 P2，见第 7 节）。

---

## 2. 范围

| 纳入 | 不纳入（本版） |
|------|----------------|
| JSONL → **人物卡库**导入（幂等、跳过/更新报告） | 从正文自动识别并推荐人物 |
| 人物卡库列表/搜索 API（创建故事时多选来源） | 与参考库 `system-corpus-reference` 的自动关联 |
| 故事下 **人物快照**：创建时从库克隆、**可编辑**、可排序、可删除、可**中途新增** | 多租户权限与人物卡可见范围（默认单库单用户） |
| 生成 Prompt 中「人物设定」区块（内容 = 该故事全部快照格式化结果） | 母卡「改一处全局同步到所有故事」（快照模型下**不承诺**） |

## 3. 用户故事

1. **作为**运营/作者，**我希望**把 `character_cards.jsonl` 导入人物卡库，**以便**复用与检索。  
2. **作为**作者，**我希望**新建故事时多选库中人物并生成快照，**以便**在本故事里改人设而不影响库。  
3. **作为**作者，**我希望**写到一半能新增或改故事内人物快照，**以便**后续生成都按最新设定。  
4. **作为**作者，**我希望**生成时快照与 RAG 同屏进 Prompt，**以便**文风与人设兼顾。

## 4. 数据契约（与现有 JSONL 对齐）

单行 JSON 顶层字段（已实现）：

| 字段 | 说明 |
|------|------|
| `characters` | 数组，元素为单角色对象 |
| `schema_version` | 人物卡 schema 版本（如 `"1"`） |
| `source_relative_path` | 来源相对路径，可追溯 |
| `content_sha256` | 来源内容哈希，可用于幂等导入 |

单角色对象字段（示例，以实际 extractor 为准）：

`name`, `age`, `identity`, `appearance`, `personality`, `background`, `triggers`, `nsfw_profile`, `psychology_and_relations`, `sample_dialogues` 等。

**已确认**：同一 `name`、不同 `source_relative_path`（及行内角色位置）→ **不同库卡**，导入主键必须能区分（推荐 UUID 或 `content_sha256 + 数组下标` 等稳定组合）。

**仍待产品定规则**：`name` 为空的角色 — 跳过 / 占位名 / 仅进 skipped 日志。

---

## 5. 故事侧：快照模型（已定稿）

**概念**：生成与 Prompt **只读「故事—人物快照」表**，不直接读人物卡库（库仅用于克隆来源与浏览）。

| 行为 | 说明 |
|------|------|
| 创建故事并选人 | 对每个 `library_character_id` **深拷贝** payload 到 `erotica_story_character_snapshots`（命名以实现为准）；可选记录 `cloned_from_library_id` 便于展示「来自某库卡」，但**不建立外键级联更新**。 |
| 编辑 | 仅更新快照行字段（名称、性格、背景等）；**母卡不变**；已写章节正文不自动回写。 |
| 新增 | 插入快照行，`cloned_from_library_id = NULL`，payload 由用户填写。 |
| 删除快照 | 从本故事移除该人设；**不影响**人物卡库。 |
| 对已有章节的影响 | 默认约定：**仅影响后续生成**；不在本需求中做「历史章节按新快照重算」。 |

创建故事 API：可传 `libraryCharacterIds: string[]`；服务端创建故事后立即插入对应快照行。

故事详情 API：返回本故事**快照列表**（含排序），供编辑页与生成侧使用。

---

## 6. 与 RAG / Prompt 的关系（需求层表述）

- **RAG**：继续负责从当前故事文档 +（可选）参考库召回**正文片段**，逻辑不变。  
- **人物性格**：从 DB 读取该故事下**全部快照**的当前内容，格式化后写入 Prompt（例如 `{{characterProfiles}}`），与 `{{ragContext}}` 并列。  
- **可选增强（P2）**：将快照摘要再写入向量库参与检索；本版**不强制**。

---

## 7. 验收标准（草案）

1. JSONL 导入人物卡库后条数与策略一致；同名不同源为多条库记录。  
2. 创建故事并传库卡 ID 后，故事下快照内容与克隆时刻一致；修改快照后，**库中母卡不变**。  
3. 故事内新增快照后，下一次生成 Prompt 中含该人物。  
4. 生成用 Prompt 中人物块与**快照表当前数据**一致，且与 RAG 块共存。  
5. 无快照时人物块为空或占位符（模板配置）。

---

## 8. 非功能需求

- 导入大批量 JSONL 时宜支持异步任务或命令行，避免 HTTP 超时。  
- `schema_version` 升级时要有迁移或兼容策略说明。
