# 项目架构设计

**文档版本**：1.1  
**更新日期**：2026-03-28  
**项目名称**：EroticaForge（本地 NSFW 角色扮演小说生成工具）

本文描述**目标分层与数据流**；包名、类名与目录树以 [`项目目录结构说明.md`](项目目录结构说明.md) 为准。

---

## 1. 产品与技术前提

- **全本地**：主生成走 **llama.cpp**（OpenAI 兼容 HTTP，通常 `llama-server`）；嵌入走 **Ollama + bge-m3**。  
- **持久化**：业务与 RAG 元数据在 **PostgreSQL**；向量存 **pgvector**（与 LangChain4j `EmbeddingStore` 集成）。  
- **主后端**：Java 21 + Spring Boot 3 + LangChain4j；持久化当前为 **MyBatis-Plus**（非 JPA）。  
- **数据分析批处理**：独立模块 `dataAnalysisModule/`，与主应用进程与端口分离。

---

## 2. 分层架构（与实现对齐）

```text
com.eroticaforge
├── EroticaForgeApplication
├── config/                    # Bean：ChatModel、EmbeddingModel、EmbeddingStore、业务配置绑定
├── domain/                    # StoryState、Story、Chapter 等纯领域模型与领域异常
├── application/
│   ├── service/               # 用例：生成、RAG 摄入/检索、状态、后处理、参考库导入等
│   └── dto/                   # API 与内部传输对象
├── infrastructure/persistence/  # MyBatis-Plus：实体、Mapper、Repository、写库门面
├── presentation/
│   ├── controller/            # REST（SSE 用于流式生成）
│   └── advice/                # 统一异常 → ApiErrorResponse / SSE error
└── utils/                     # Prompt 模板、语料路径工具等
```

### 依赖方向

1. **domain** 不依赖 Spring Web 与持久化细节。  
2. **application** 编排领域逻辑与基础设施；对外由 presentation 调用。  
3. **infrastructure** 实现持久化与外部资源访问。  
4. **presentation** 只做 HTTP 适配与校验，核心业务在 application。

## 3. RAG 与生成数据流（概要）

### 3.1 摄入（Ingestion）

用户上传 **.txt** → `RagIngestionService`：切分（`RagProperties` 控制块长与重叠）→ `EmbeddingModel` → `EmbeddingStore` 写入；文档元数据写入业务表（如 `erotica_documents`）。  

专题参考库：数据分析模块产出 JSONL 后，由 `CorpusJsonlReferenceImporter` 按行解析路径与元数据，写入固定 `story_id`（默认 `system-corpus-reference`），与主应用故事向量**同一套**存储与 metadata 约定。

### 3.2 检索（Retrieval）

`RagRetrievalService`：用查询（可选拼接当前摘要）做嵌入 → 向量检索 → 按 `story_id` 等在内存侧过滤（必要时过取再裁剪）→ 格式化为带标签上下文字符串。  

若 `erotica.rag.include-reference-corpus=true`，参考库切块与当前故事切块一并参与检索与排序。

### 3.3 生成（Generation）

`NovelGenerationService`：拼装系统/用户 Prompt（含 RAG 上下文、StoryState、Lorebook 等）→ 调用流式或阻塞 **ChatModel** → `PostGenerationService` 持久化章节并更新 StoryState（可选后续 RAG 再索引由配置控制）。

---

## 4. 故事一致性与防循环（原则）

- 以 **StoryState**（摘要、人物状态、事实、标记等）压缩长期记忆，避免全文塞进上下文。  
- 每轮生成后由 **PostGenerationService** 闭环写库；状态更新需考虑 **乐观锁**（`version`）。  
- 采样与 Prompt 规则见 [`核心 Service 实现指南.md`](../development/核心%20Service%20实现指南.md) 与 PRD。

---

## 5. 接口形态（实现现状）

- **REST + SSE** 满足写作与调试；**未**采用 WebSocket 作为默认方案。  
- 对外路径与语义见 [`API 接口定义.md`](../api/API%20接口定义.md)。  
- 默认 HTTP 端口 **8090**，避免与常见 llama-server 端口冲突（可配置）。

---

## 6. 可扩展点

- 多模型链：`useMultiModel` 与 `多模型协作.md` 规划一致，实现后在不破坏 API 的前提下接入。  
- Lorebook：当前内存实现可替换为表存储，与数据模型文档对齐。  
- 参考库：为 `CorpusJsonlReferenceImporter` 增加运维入口（CLI/REST）与按标签过滤策略。  
- 观测：日志字段（`storyId`、耗时）；可按需引入 Micrometer。
