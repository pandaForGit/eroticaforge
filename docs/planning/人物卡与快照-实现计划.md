# 人物卡库 + 故事快照 — 实现计划

**版本**：1.1  
**日期**：2026-04-03  
**前置结论**（已确认）：同名不同源分卡；创建故事后可改绑定；性格 = DB 快照 + 模板分节（与 RAG 并列）。

**关联需求/架构**：`docs/product/人物卡导入与故事绑定需求说明.md`、`docs/architecture/人物卡与RAG编排业务流程.md`

**实施报告**（落地说明与验证步骤）：`docs/planning/人物卡与快照-实施报告.md`

---

## 阶段 0：契约冻结

- 冻结 JSONL 行结构（已有 `characters[]`、`content_sha256`、`source_relative_path`、`schema_version`）。  
- 定稿库表唯一键：`content_sha256` + `role_index`，与架构文档 §6 一致。  
- 定稿快照表字段与 API 错误码（非法 `libraryCharacterId` → 400）。

---

## 阶段 1：数据库与领域层

1. 新增 SQL 迁移 `sql/002_character_library_and_snapshots.sql`：`erotica_character_library`、`erotica_story_character_snapshots`（`story_id` 级联删除）。  
2. MyBatis-Plus：实体、Mapper、`CharacterLibraryRepository`、`StoryCharacterSnapshotRepository`。

---

## 阶段 2：JSONL 导入人物卡库

1. `CharacterCardsJsonlImporter` + `CharacterCardsJsonlImportApplicationRunner`，路径 `erotica.character-cards-import`。  
2. 逐行解析：展开 `characters[]`，`role_index` = 数组下标；按 `content_sha256`+`role_index` 幂等 upsert。  
3. 日志：inserted / updated / skipped / failed。

---

## 阶段 3：REST API

| 能力 | 端点 |
|------|------|
| 库列表/搜索 | `GET /api/character-library?query=&limit=` |
| 创建故事 + 可选克隆 | `POST /api/stories`：`libraryCharacterIds[]` |
| 故事下快照 | `GET/POST /api/stories/{id}/character-snapshots` |
| 改快照 | `PATCH .../character-snapshots/{snapshotId}` |
| 删快照 / 调序 | `DELETE` + `PUT .../character-snapshots/order` |

详见 `docs/api/API 接口定义.md` §3.2。

---

## 阶段 4：生成链路接入

1. `NovelGenerationService`：`StoryCharacterSnapshotService.buildProfilesTextForPrompt(storyId)`。  
2. `PromptComposer#buildFullPrompt`：`characterProfiles` + 模板 `{{characterProfiles}}`。  
3. `PromptProperties` / `application.yml` 注释同步占位符列表。

---

## 阶段 5：前端

1. 新建故事：库列表多选 + `libraryCharacterIds`。  
2. 故事详情：快照列表、从库添加、空白卡、JSON 编辑、删除、上移/下移。

---

## 阶段 6：收尾

- 端到端：导入 → 建故事选人 → 改快照 → 生成核对 Prompt。
